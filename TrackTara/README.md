# 📦 TrackTara - Система управління контейнерами та продуктами

TrackTara - це веб-додаток для управління контейнерами та продуктами з підтримкою різних ролей користувачів та повним функціоналом CRUD операцій.

## 🚀 Швидкий старт

### Встановлення залежностей

```bash
npm install
```

### Налаштування

Скопіюйте шаблон у `.env` у корені проекту (PowerShell: `Copy-Item .env.example .env`; bash: `cp .env.example .env`).

Мінімум для демо на моках:

```env
VITE_USE_MOCK_API=true
```

Вхід через **Google**: додайте Client ID з Google Cloud Console (тип **Web application**):

```env
VITE_GOOGLE_CLIENT_ID=xxxxxxxx.apps.googleusercontent.com
```

На **Vercel** задайте ту саму змінну в Settings → Environment Variables і перезбереріть проєкт. У GCP для цього Client ID вкажіть **Authorized JavaScript origins**: `http://localhost:5173` та URL продакшену (`https://….vercel.app`). Поки застосунок у статусі **Testing**, додайте свій Gmail у **Test users** на екрані згоди OAuth.

### Запуск проекту

```bash
npm run dev
```

Проект буде доступний за адресою: `http://localhost:5173`

## 🌐 Деплой на Vercel (без бекенду, лише моки)

1. Залийте репозиторій на GitHub і підключіть його у Vercel («Import Git Repository»).
2. Vercel автоматично виявить **Vite** (build: `npm run build`, output: `dist`).
3. У **Settings → Environment Variables** задайте:
   - `VITE_USE_MOCK_API=true` (або не задавайте — це і є дефолт)
   - `VITE_GOOGLE_CLIENT_ID=...` (опційно, для входу через Google)
4. У `vercel.json` уже додано **SPA fallback** — будь-який маршрут React Router відкривається без 404.
5. Деплой → демо живе без бекенду.

### Зберігання даних і робота в кількох вкладках

Усі мок-дані (товари, контейнери, замовлення, реєстри, історії, налаштування паролів)
лежать у спільному ключі `tt:mockdb:v1` у `localStorage`. Завдяки цьому:

- **Дані живі між F5** і навіть між сесіями браузера (на одному пристрої/профілі).
- **Зміни синхронізуються між вкладками автоматично** через стандартну подію `storage`.
- Кожна мутація моків (push/splice/змінне поле) одразу пише знімок у `localStorage`.
- Скинути дані до seed-стану: у DevTools → Application → Local Storage видалити ключ
  `tt:mockdb:v1` і оновити сторінку (або викликати `resetAllMocks()` зі сховища, якщо
  десь додасте кнопку для адміна).

> Reduxʼом керує **redux-persist** (ключ `persist:tracktara`) — для UI-стану це
> кеш на одну вкладку. Бізнес-дані синхронізуються через мок-сховище вище.

## 🔧 Mock API

Проект використовує систему Mock API для розробки без залежності від бекенду.

### Перемикання між моками та реальним API

В файлі `.env`:

```env
# Для мок-даних
VITE_USE_MOCK_API=true

# Для реального API (коли буде готовий)
# VITE_USE_MOCK_API=false
```

### Мок-дані

Моки містять початкові дані:

- 3 тестових користувачі
- 3 продукти
- 3 контейнери
- 4 типи продуктів
- 4 типи контейнерів

Дані зберігаються в пам'яті та скидаються при перезавантаженні сторінки.

**Детальна інформація:** Дивіться [MOCK_API_GUIDE.md](./MOCK_API_GUIDE.md)

## 🏗️ Архітектура

### State Management (Redux)

- **Redux Toolkit** для управління станом
- **Redux Thunk** для асинхронних actions
- Розділення на slices за доменами:
  - `userSlice` - поточний користувач
  - `usersSlice` - список користувачів
  - `productSlice` - продукти
  - `containerSlice` - контейнери
  - `containerTypeSlice` - типи контейнерів
  - `productTypeSlice` - типи продуктів
  - `appSettingSlice` - глобальні налаштування
  - та інші...

### Маршрутизація

- **React Router v7** для навігації
- Захищені маршрути через `ProtectedRoute`
- Перевірка ролей для доступу до сторінок

### Сервіси

- **ServiceFactory** автоматично вибирає між моками та реальним API
- Всі сервіси мають однакові інтерфейси
- Централізований експорт через `utils/services/index.js`

### HTTP клієнт

- **HttpClient** - обгортка над Axios
- Автоматичне додавання токенів
- Interceptor для оновлення токенів
- Глобальний loading state

## 🐛 Troubleshooting

### Помилки при запуску

1. **Помилка з імпортами сервісів**

   - Перевірте що використовуєте `import { Service } from '@/utils/services'`
   - Не використовуйте прямі імпорти з `services/ServiceName`

2. **Моки не працюють**

   - Перевірте наявність `.env` файлу
   - Перевірте що `VITE_USE_MOCK_API=true`
   - Перезапустіть dev сервер

3. **Помилки з токенами**
   - Очистіть localStorage: `localStorage.clear()`
   - Спробуйте увійти знову

### Консоль браузера

При успішному запуску з моками ви побачите:

```
🔧 [SERVICE FACTORY] Using MOCK services
```
